Design System One - Mobile

An ongoing project following atomic design patterns with elements inspired by Samsung's One UI Design Guidelines. I am creating a design system for long-term skill progression and component availability for future projects. The design system will be referred to as Design System One.

Kevin Towner•5 minute read

Introduction

Why create a design system?

Design systems on-hand are productive tools when it comes to creating consistent and reusable design patterns for user interfaces. Design systems can set branding and identity standards as well as drive realistic goals.

Benefits

  • • Component reusability
  • • Design consistency
  • • UI communication

Brief

Design System One consists of visual elements, a growing library inspired by One UI Design Guidelines, and Atomic Design patterns. The UI library is currently comprised of primarily Settings-based components.

First Iteration

Summary

Iteration 1 was a great starting point for building Design System One and allowed me to experiment with visual elements such as color, type, and fonts. It also laid a solid foundation for building screens for menus and/or listed content. Iteration 1 established fixed vs scrolled content and how to layer components correctly to mimick the functions of a mobile Samsung Galaxy product.

Design System One was far from perfect. There was a lack of size and scope to the UI Library, and there were areas in which Design System One could be more flexible (menu-item responsiveness). Components and visual elements are not as organized as they should be which has led to inconsistencies in naming conventions, component sizing constraints, and page level redundancies.

Visual & Component Libraries

Iteration 1

First iteration had 7 pages of visual design elements and user interface components. 4 of the pages were dedicated to visual elements while 3 of the pages were dedicated to user interface components. The overabundance of visual design pages was due to redunancies in visual element locations.

Iteration 1's figma pages housing atomic design methodology. Iteration 2's figma pages housing atomic design methodology.

Iteration 2

I've reduced the amount of pages in Design System One to 3 - not including 1 page serving as a test canvas, or a "workbench". The "ui library" page houses atom, molecule, organism, and template level components. "Screens" page are where final screens are placed and prototyped. "Visual" page houses fonts, icons, colors, and images.

Menu Items

Properties

Menu-item content margins from Iteration 1 were disproportionate and took away from the refinement of the design system (top: 22px, bottom: 22px, left: 24px). Menu-items have been redesigned to have equal content margins of 24px (top, bottom & left).

Resizing

Menu-item sizing and constraints from Iteration 1 did not account for the possibility of menu-item content labels to have multiple lines of text. For Iteration 2, I adjusted the resizing of the element's container to maintain its 24px consistency with the total height of item content.

Drawback 1

Background (container) is responsive to menu-item content's height while the actual component dimensions are not. Components can be manually resized to accomdate the height of the container without affecting menu-item content dimensions, but this should be taken into consideration for following iterations.

Drawback 2

I currently have the width of the text content extending beyond the (previously hidden) toggle inputs on the right side of the menu-item. Having this drawback takes flexibility away from the design system. Currently, most menu-items do not need toggles which is why, for the time being, I opted to widen the length of the text labels.
